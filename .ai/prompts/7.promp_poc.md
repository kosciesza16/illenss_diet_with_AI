Cel PoC — krótkie streszczenie: Stwórz Proof of Concept, który weryfikuje kluczową funkcjonalność aplikacji Personalized Recipe Assistant: mapowanie składników przepisu na podstawowe wartości odżywcze i prostą
AI napędzaną modyfikację przepisu (substytucje składników i skalowanie porcji) dostosowaną do jednego z trzech docelowych schorzeń. 
PoC ma być minimalny, szybki do wykonania i pokazowy — tylko to, co konieczne do potwierdzenia koncepcji.
Kontekst techniczny:
Preferowane technologie dla PoC (rekomendacja): Backend minimalny - Supabase Auth (OAuth) + PostgreSQL (JSONB) Frontend minimalny - React, Astro, TypeScript.
AI: użyj usługi Openrouter.ai (dostęp do modeli)
Dane: trzymamy mapping ingredient→nutrient w lokalnej tabeli(mały dataset do PoC).
Wymagania PoC (konkretne, testowalne):
Endpoints API (backend minimalny):
POST /recipes
przyjmuje JSON: { title, servings, ingredients: [{ name, quantity, unit }], steps: [string] }
walidacja: title non-empty, każdy step 10–500 znaków, każdy ingredient ma name+quantity+unit
przechowuje surowy tekst i JSONB w Postgres
GET /recipes/:id
zwraca surowy tekst i strukturę JSONB
GET /recipes (lista dla zalogowanego usera)
POST /ai/modify
wejście: { recipeId | recipePayload, operation: "substitute"|"scale"|"map_nutrients", userProfile: { disease, age, sex, allergies[] } }
odpowiedź (synchronous dla prostych operacji): zmodyfikowany przepis w strukturze JSON oraz agregowane wartości odżywcze per porcję i % dziennego celu (zgodnie z profilem) — dla
PoC obsłużyć tylko prostą mapę i proste substytucje
